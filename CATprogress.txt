 1. EPIC SUMMARY
================================================================================

Goal (from your epic description):
  InfoPharma maintains an up-to-date electronic catalogue, manages stock
  levels, and monitors inventory thresholds so products remain available.

Current state (high level):
  • Full product CRUD implemented: create (US2), update (US4), delete with
    audit log (US3), multi-criteria search (US5), merchant stock masking (US6),
    stock delivery recording (US7), and minimum stock thresholds (US8).
  • Remaining: automated low-stock warnings (US9, partial — UI warning shown
    in admin table when stock ≤ threshold) and low-stock reporting (US10).

Estimated completion for CAT epic (prototype):  Most stories require new
backend endpoints, entity fields, validation, audit/logging, and frontend
screens — see per-story sections below.

 CAT-US1 — Catalogue Initialization
──────────────────────────────────────────────────────────────────────────────

Title:
  As an Administrator, I want to create a new electronic catalogue (Database)
  so that pharmaceutical products can be stored and managed in the system.

ACCEPTANCE CRITERION 1:
  "The system must allow the creation of a new catalogue only if one does
   not exist already."

  STATUS: ⚠️  INTERPRETATION-DEPENDENT / MOSTLY IMPLICIT

  DONE TODAY:
    • The application uses a single JPA entity mapped to table "products".
      Hibernate ddl-auto creates the table when the app starts against an
      empty database — there is no second "catalogue" to duplicate in the
      relational model.

  NOT DONE / GAP:
    • There is NO explicit administrator workflow or API such as
      "POST /api/catalogue/initialize" that checks "catalogue exists?" and
      returns 409 if already present.
    • If your marker expects a visible "create catalogue" action in the UI
      or a dedicated Catalogue aggregate (separate from Product rows), that
      is NOT implemented.

  WORK TO COMPLETE (if strict compliance needed):
    • Decide model: single implicit catalogue (document as acceptable per
      brief footnote: optional catalogue creation) OR add a catalogue
      metadata table (e.g. catalogues: id, name, created_at) with at-most-one
      active row enforced by app logic or DB constraint.
    • If required: Admin-only endpoint + optional UI "Initialize catalogue"
      that fails when products table already has rows or when catalogue
      record exists.


ACCEPTANCE CRITERION 2:
  "The system must initialize the catalogue with the appropriate fields such
   as Product ID, Description, Unit Price, and Availability with the correct
   data types (e.g. int, String)."

  STATUS: ⚠️  PARTIAL — FIELD MAPPING DIFFERS FROM WORDING

  DONE TODAY:
    • description  → String  ✅
    • unit price   → BigDecimal (mapped to DECIMAL in DB) ✅  (preferred over
                     double for money; acceptable vs "numeric type")
    • availability → Integer (availabilityCount) ✅

  GAPS VS STORY WORDING:
    • "Product ID" in the story often implies a business identifier (SKU)
      separate from the database primary key.  Today the only identifier is
      id: Long, auto-generated by the database — you cannot assign a custom
      product code at creation time via the API as specified in US2/US4.

  WORK TO COMPLETE:
    • Add optional or required productCode / sku field (String) with UNIQUE
      constraint, OR document that surrogate Long id is the Product ID for
      the prototype (and align US2 "unique Product ID" with that).
    • Add Jakarta Bean Validation on DTOs (@NotBlank, @DecimalMin, @Min(0))
      aligned with US2/US4.

  "The system must prompt for a confirmation Yes/No before a product is
   deleted."

  STATUS: ✅  DONE

ACCEPTANCE CRITERION 2:
  "The system must log the deletion activity, including which user performed
   the action."

  STATUS: ✅  DONE

  IMPLEMENTED:

    • ProductService.deleteProduct() saves the log and deletes the product

      returns 409 CONFLICT (preserves historical order integrity).
    • Mirrors the StandingChangeLog pattern from ACC-US5.
