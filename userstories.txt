User Stories IPOS – SA System
Overview of the User Stories: All of the user stories below are related to the IPOS-SA system 
and gathered from the required functionality provided in the Brief. Each User Story follows 
the Industry standard User Story structure of Title -> Description (Optional) -> Acceptance 
Criteria. All of these User Stories are created from the Users perspective when using the 
system. Furthermore, the Acceptance Criteria has been made in a way that it is either a pass 
or fail metric for each point. The main goal of these user stories was to provide a easy 
transition to turning these user stories into Tickets (tasks to complete) which can be done 
within a Sprint (2 weeks) and can be assigned to a developer on a platform such as GitHub 
Issues or JIRA. Also, each Epic of the System has been broken down into the User Stories as 
shown below. Finally, each user story has met the INVEST framework.
Epic: User Account Management (IPOS – SA – ACC)
Description: This epic covers the creation, configuration, and access levels of different user roles 
within the IPOS-SA system to ensure secure and tailored access for merchants, administrators, and 
managers.
User Story 1: Merchant Account Creation:
- Title: As an Administrator, I want to create a new Merchant account which requires contact 
and financial details so that the account is properly configured before it becomes active. 
- Description: Specific metadata is required to create a Merchant Account (Contact details, 
Credit Limit, and Discount Plan for example). 
- Acceptance Criteria (Must be yes or no): 
o System must provide fields for Contact Details, Credit Limit, and Discount Plan 
selection.
o The account must remain "Inactive" and cannot be created if any of the mandatory 
fields are left blank.
o Once all details are saved, the account status should update to "Active."
User Story 2: Fixed Discount Plan Assignment:
- Title: As an Administrator, I want to assign a fixed discount rate to a Merchant so that a 
consistent discount is applied to every order they place. 
- Acceptance Criteria: 
o The system must allow the user (Administrator) to input a specific percentage rate for 
the "Fixed" plan.
o The system must automatically reduce the "Amount Due" for every order placed by 
this merchant based on that fixed rate.
User Story 3: Flexible Discount Plan Configuration:
- Title: As a Manager, I want to set up a flexible discount plan based on monthly spend so that 
merchants have a incentive to spend more.
- Acceptance Criteria: 
o The system must track the total value of all orders within a single calendar month.
o The system must apply tiered rates (e.g. 1% for <£1k, 2% for £1k-£2k, 3% for >£2k).
o At the end of the month, the system must calculate the total discount to be either 
refunded via check or credited to the next order.
User Story 4: Role-Based Access Control (RBAC):
- Title: As an Administrator, I want to assign specific roles (Merchant, Admin, Manager) to users 
so that they can only access the relevant functionality that they require. 
- Acceptance Criteria: 
o Merchant accounts must only have access to IPOS-SA-CAT (Catalogue) and IPOS-SAORD (Orders).
o Manager accounts must have access to IPOS-SA-RPRT (Reporting) and Merchant 
account settings.
o Administrator accounts must have full access to all packages, including IPOS-SA-ACC.
User Story 5: Managing Defaulted Accounts:
- Title: As a Manager, I want to manually restore a merchant's account from 'In Default' status 
so that they can access the functionality of their account again after a long-term payment 
issue (e.g longer than 30 days without payment) is resolved. 
- Acceptance Criteria: 
o The system must restrict the "Restore" action for "In Default" accounts (delays >30 
days) to Manager-level users only.
o A Manager must be able to change the status from "In Default" to either "Normal" or 
"Suspended."
o The system must log which Manager performed the status change. 
User Story 6: Managing Accounts:
- Title: As a Manager, I want to be able to access merchant accounts and alter their credit limits, 
discount plans.
- Acceptance Criteria:
o The system must allow a manager to access merchant accounts.
o The system must allow a manager to alter the credit limits and discount plans of the
merchant accounts (once logged in).
Epic: Electronic Catalogue and Inventory Control
Description: This epic enables InfoPharma to maintain an up-to-date electronic catalogue of 
pharmaceutical products, manage stock levels, and monitor inventory thresholds to ensure products 
are available to the customers.
User Story 1: Catalogue Initialization
- Title: As an Administrator, I want to create a new electronic catalogue (Database) so that 
pharmaceutical products can be stored and managed in the system.
- Acceptance Criteria:
o The system must allow the creation of a new catalogue only if one does not exist
already.
o The system must initialize the catalogue with the appropriate fields such as Product ID, 
Description, Unit Price, and Availability with the correct data types (e.g int, String).
User Story 2: Product Creation
- Title: As an Administrator, I want to add new products to the catalogue so that they are 
available for merchants to order.
- Description: Requires Product ID, Description, Unit Price, and Availability.
- Acceptance Criteria:
o The system must enforce a unique Product ID for every new entry.
o The system must validate that the Unit Price is greater than zero.
o The system must validate that Availability is not a negative value.
User Story 3: Product Discontinuation/Removal
- Title: As an Administrator, I want to remove products from the catalogue so that merchants 
cannot order discontinued or out of stock items.
- Acceptance Criteria:
o The system must prompt for a confirmation "Yes/No" before a product is deleted. (This 
is to prevent accidental deletion.)
o The system must log the deletion activity, including which user performed the action.
User Story 4: Product Data Maintenance
- Title: As an Administrator, I want to update existing product details so that the catalogue 
information remains up to date and accurate for merchants.
- Acceptance Criteria:
o The system must prevent the modification of an existing Product ID.
o The system must validate that any updated price or stock level (Availability) remains 
non-negative.
User Story 5: Internal Product Search
- Title: As an Administrator, I want to search for products using multiple criteria so that I can 
quickly locate products and see all of the information relating to the product.
- Acceptance Criteria:
o The system must allow searching by Product ID, Description, or Price Range.
o The system must display a "No products found" message if no matches exist.
User Story 6: Merchant Catalogue Browsing
- Title: As a Merchant, I want to search and view the product catalogue so that I can identify 
items I wish to purchase.
- Acceptance Criteria:
o The system must provide "Read-Only" access, the merchant cannot edit any catalogue 
details.
o The system must hide the products stock levels from the merchant view (only showing 
"Available" or "Out of Stock").
o The system must allow for products to be searched from incomplete words (para -> 
can search for paracetamol)
User Story 7: Stock Delivery Recording
- Title: As an Administrator, I want to record new stock deliveries so that inventory levels are 
automatically increased.
- Acceptance Criteria:
o The system must record the delivery date and the quantity received.
o The system must automatically increment the "Availability" of the product by the 
delivery quantity.
o The system must reject any delivery quantity equal to or less than zero.
User Story 8: Minimum Stock Thresholds
- Title: As an Administrator, I want to set minimum stock thresholds for each item so that the 
system can flag when a product reaches low enough levels of stock.
- Acceptance Criteria:
o The system must allow a specific minimum stock threshold value to be assigned to 
each unique Product ID.
o The system must validate that threshold values are not negative.
User Story 9: Automated Low-Stock Warnings
- Title: As an Administrator, I want to receive automated UI warnings when stock falls below 
thresholds set so that I can trigger the re-ordering process to replenish the stock.
- Acceptance Criteria:
o The system must automatically flag a product in the UI when Current Availability < 
Threshold.
o The warning must include the Product ID, Description and current stock level.
User Story 10: Low-Stock Reporting
- Title: As a Manager, I want to generate a low-stock report so that I can oversee inventory 
trends and plan supplier orders.
- Acceptance Criteria:
o The system must generate a report listing all products currently below their defined 
thresholds.
o The report must reflect real-time catalogue data at the moment of generation.
o The report must be accessible through the IPOS-SA-RPRT package.
Epic: Order Fulfillment and Financial Tracking
Description: This epic enables merchants to place and track orders while providing InfoPharma with 
the tools to manage stock reduction, invoice generation and payment recording. 
User Story 1: Placing a New Order
- Title: As a Merchant, I want to place an order by selecting items from the current catalogue so 
that I can restock my pharmacy. 
- Description: Dependency: Requires integration with the catalogue (IPOS-SA-CAT).
- Acceptance Criteria:
o The system must verify the merchant is authorized and authenticated before allowing 
order entry (That they have a Merchant Account). 
o The merchant must be able to select one or multiple items and specify valid quantities. 
o Upon submission, the order must be successfully recorded in the database with an 
initial status (e.g "Accepted").
User Story 2: Real-time Order Tracking
- Title: As a Merchant, I want to track the progress of my submitted orders so that I can monitor 
when they are processed or dispatched. 
- Acceptance Criteria:
o The merchant must see a list of their orders with current statuses (e.g "Accepted" 
"Processing" "Dispatched").
o Status updates must be reflected in the UI as they are updated by staff or automated 
processes.
User Story 3: Financial Balance Oversight
- Title: As a Merchant, I want to view my current outstanding balance so that I know the total 
amount I need to repay InfoPharma. 
- Acceptance Criteria:
o The system must display a read-only total of all unpaid invoices for the merchant. 
o The balance must automatically update when new invoices are raised or payments are 
recorded.
o The system must display amount of time elapsed since payment has been due.
User Story 4: Automated Stock Reduction
- Title: As a Merchant, I want the system to automatically reduce product availability upon 
order acceptance so that I do not accidentally order items that have already been sold to 
others.
- Description: This story ensures the inventory data remains synchronized with successful sales 
transactions without manual intervention.
- Acceptance Criteria:
o Stock levels for specific items must be decremented only after the order status moves 
to "Accepted" 
o The reduction quantity must exactly match the quantity specified in the accepted 
order. 
User Story 5: Automated Invoice Generation
- Title: As an Administrator, I want the system to generate an invoice for every accepted order 
so that billing and payments can be tracked. 
- Acceptance Criteria:
o The system must automatically generate an invoice once an order is marked as 
"Accepted" 
o The invoice must include Merchant contact details, order items, VAT number, and final 
total values.
o Invoices must be stored in the system database for future reference. 
User Story 6: Recording Merchant Payments
- Title: As an Administrator, I want to manually record payments against existing invoices so that 
outstanding balances are kept up to date. 
- Acceptance Criteria:
o The user must be able to record payments made via Bank Transfer, Card or Check. 
o The system must update the merchant’s outstanding balance immediately upon saving 
the payment.
Epic: Operational and Financial Reporting
Description: This epic provides management and staff with tools to analyze sales performance, 
merchant activity and stock turnover through on-screen visualization and printed reports.
User Story 1: Sales Turnover Reporting
- Title: As a Manager, I want to generate a turnover report for a specific period so that I can 
analyse company sales performance. 
- Acceptance Criteria:
o The system must allow users to select a specific "Start Date" and "End Date." 
o The report must display total quantities sold and the total revenue received. 
o The report must be viewable on-screen and formatted for printing. 
User Story 2: Merchant History Tracking
- Title: As a Manager, I want to generate an order history report for a specific merchant so that I 
can review their purchase/order history. 
- Acceptance Criteria:
o The report must include: Order ID, Order Date, Dispatch Date, Total Value and 
Payment Status (e.g Pending). 
o The system must include a "Totals" line at the end of the report to summarize the 
period.
User Story 3: Individual Merchant Activity Report
- Title: As a Manager, I want to generate a detailed activity report for a merchant so that I can 
review their ordering behavior. 
- Acceptance Criteria:
o The report header must display the merchant's contact details. 
o Each order in the list must show: individual items, quantities, item costs, total order 
cost and any applied discounts. 
User Story 4: Global Invoice Monitoring
- Title: As a Manager, I want to view all invoices raised against all merchants for a given period 
so that I can monitor company-wide billing activity. 
- Acceptance Criteria:
o The report must list every invoice issued within the user-defined date range across the 
entire system. 
o Each entry must include the Merchant Identifier, Invoice ID, Date, Amount, and 
Payment Status.
User Story 5: Stock Turnover Analysis
- Title: As a Manager, I want to generate a stock turnover report so that I can compare products
sold against products newly received. 
- Description: Requires accurate data from both IPOS-SA-ORD and IPOS-SA-CAT ensure these 
systems are operational before tackling this.
- Acceptance Criteria:
o The report must display quantities of products sold versus quantities of products
received into stock for the selected period. 
o Data must be accurately pulled from both the IPOS-SA-ORD (sales) and IPOS-SA-CAT 
(deliveries) records.
