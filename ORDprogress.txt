 1. EPIC SUMMARY
================================================================================

Goal (from your epic description in userstories.txt):
  This epic enables merchants to place and track orders while providing
  InfoPharma with the tools to manage stock reduction, invoice generation
  and payment recording.

Current state (high level):
  - ORD-US1 COMPLETE: POST /api/orders -> OrderService.placeOrder()
    (transactional stock decrement, price snapshot, FIXED/FLEXIBLE discounts,
    credit limit). Merchant isolation enforced at service layer. New orders
    are created with status ACCEPTED (aligned with brief wording).
    Frontend supports multi-line item selection. Empty items rejected
    with 400.
  - ORD-US2 COMPLETE: GET /api/orders returns role-scoped results
    (MERCHANT sees own orders only; MANAGER/ADMIN see all).
    PUT /api/orders/{id}/status allows MANAGER/ADMIN to advance the
    lifecycle: ACCEPTED -> PROCESSING -> DISPATCHED, with cancellation from
    ACCEPTED or PROCESSING. Frontend shows "My Orders" / "All Orders" table
    with status badges, staff action buttons, and 15-second auto-polling.
  - Order status enum: ACCEPTED, PROCESSING, DISPATCHED, CANCELLED
    (legacy PENDING/CONFIRMED kept with @Deprecated for DB compatibility;
    migration SQL documented in Order.java).
  - Stock is decremented atomically during order placement (status is
    ACCEPTED at save time), satisfying ORD-US4 criterion that stock decrements
    happen when status reaches "Accepted".
  - ORD-US5 COMPLETE: Invoice entity with InvoiceLine children, auto-generated
    in the same @Transactional boundary as order placement. Invoices snapshot
    merchant contact details, VAT number, order items, and financial totals.
    Human-readable invoice number (INV-YYYY-NNNNN). 1:1 with Order (unique FK).
    Frontend Invoices page shows role-scoped listing and invoice detail view.
  - ORD-US6 COMPLETE: Payment entity with method enum (BANK_TRANSFER, CARD,
    CHEQUE). POST /api/invoices/{id}/payments (ADMIN only) records payments
    against invoices with amount validation (must not exceed outstanding).
    Frontend admin payment recording form on Invoices page.
  - ORD-US3 COMPLETE: GET /api/merchant-financials/balance (MERCHANT only)
    returns outstanding total (sum invoices - sum payments), currency, oldest
    unpaid due date, and days elapsed since due. Balance is derived (no stored
    field) so it auto-updates when invoices or payments change.
    Frontend balance card on Invoices page for MERCHANT users.
  - MerchantProfile extended with vatRegistrationNumber and paymentTermsDays
    (default 30). Invoice dueDate = issuedAt + paymentTermsDays.
  - Credit limit check uses net exposure: sum(non-cancelled order totalDue)
    minus sum(payments for that merchant), floored at zero, then plus the
    new order's totalDue — aligned with recorded payments (ORD-US6).
  - Frontend: OrderForm.jsx supports multi-line order placement with
    add/remove item rows, an orders tracking table with role-based columns,
    staff status controls, and auto-refresh polling.
  - Frontend: Invoices.jsx provides invoice list, detail view, balance summary,
    and admin payment recording.

Epic: Order Fulfillment and Financial Tracking
Description: This epic enables merchants to place and track orders while providing InfoPharma with 
the tools to manage stock reduction, invoice generation and payment recording. 
User Story 1: Placing a New Order
- Title: As a Merchant, I want to place an order by selecting items from the current catalogue so 
that I can restock my pharmacy. 
- Description: Dependency: Requires integration with the catalogue (IPOS-SA-CAT).
- Acceptance Criteria:
o The system must verify the merchant is authorized and authenticated before allowing 
order entry (That they have a Merchant Account). 
o The merchant must be able to select one or multiple items and specify valid quantities. 
o Upon submission, the order must be successfully recorded in the database with an 
initial status (e.g "Accepted").
User Story 2: Real-time Order Tracking
- Title: As a Merchant, I want to track the progress of my submitted orders so that I can monitor 
when they are processed or dispatched. 
- Acceptance Criteria:
o The merchant must see a list of their orders with current statuses (e.g "Accepted" 
"Processing" "Dispatched").
o Status updates must be reflected in the UI as they are updated by staff or automated 
processes.
User Story 3: Financial Balance Oversight
- Title: As a Merchant, I want to view my current outstanding balance so that I know the total 
amount I need to repay InfoPharma. 
- Acceptance Criteria:
o The system must display a read-only total of all unpaid invoices for the merchant. 
o The balance must automatically update when new invoices are raised or payments are 
recorded.
o The system must display amount of time elapsed since payment has been due.
User Story 4: Automated Stock Reduction
- Title: As a Merchant, I want the system to automatically reduce product availability upon 
order acceptance so that I do not accidentally order items that have already been sold to 
others.
- Description: This story ensures the inventory data remains synchronized with successful sales 
transactions without manual intervention.
- Acceptance Criteria:
o Stock levels for specific items must be decremented only after the order status moves 
to "Accepted" 
o The reduction quantity must exactly match the quantity specified in the accepted 
order. 
User Story 5: Automated Invoice Generation
- Title: As an Administrator, I want the system to generate an invoice for every accepted order 
so that billing and payments can be tracked. 
- Acceptance Criteria:
o The system must automatically generate an invoice once an order is marked as 
"Accepted" 
o The invoice must include Merchant contact details, order items, VAT number, and final 
total values.
o Invoices must be stored in the system database for future reference. 
User Story 6: Recording Merchant Payments
- Title: As an Administrator, I want to manually record payments against existing invoices so that 
outstanding balances are kept up to date. 
- Acceptance Criteria:
o The user must be able to record payments made via Bank Transfer, Card or Check. 
o The system must update the merchant's outstanding balance immediately upon saving 
the payment.

 ORD-US1 — Placing a New Order
------------------------------------------------------------------------------

Full story text: section 4 — User Story 1.

STATUS: DONE

DONE:
  - Session authentication required for /api/orders/** (SecurityConfig).
  - Merchant must have a MerchantProfile (validated in OrderService); standing
    must be NORMAL to place orders.
  - ORD-US1 isolation: if caller role is MERCHANT, merchantId is forced to
    the caller's user ID (OrderService.placeOrder).
  - Multiple line items supported in both API and frontend UI.
  - Frontend OrderForm.jsx: dynamic add/remove item rows, product dropdown
    with stock labels, quantity inputs, minimum one line enforced.
  - Order persisted with line items and financial snapshot (totalDue, etc.).
  - Initial status set to ACCEPTED (matching the brief "e.g. Accepted").
  - Empty items list rejected with HTTP 400.
  - Credit limit: compares creditLimit to net exposure = sum(non-cancelled
    order totalDue) minus sum(payments against that merchant's invoices),
    plus this order's totalDue (payments align with ORD-US6).

MIGRATION NOTE:
  Legacy orders may have PENDING or CONFIRMED status from before the enum
  update. Run this SQL once on existing databases:
    UPDATE orders SET status = 'ACCEPTED' WHERE status IN ('PENDING','CONFIRMED');

 ORD-US2 — Real-time Order Tracking
------------------------------------------------------------------------------

Full story text: section 4 — User Story 2.

STATUS: DONE

DONE:
  - GET /api/orders returns role-scoped results:
    MERCHANT: own orders only (via findByMerchant_IdOrderByPlacedAtDesc).
    MANAGER/ADMIN: all orders (via findAllByOrderByPlacedAtDesc).
  - Order status lifecycle: ACCEPTED -> PROCESSING -> DISPATCHED (forward),
    ACCEPTED | PROCESSING -> CANCELLED (cancellation branch).
    DISPATCHED and CANCELLED are terminal states.
  - PUT /api/orders/{id}/status: MANAGER/ADMIN only (SecurityConfig URL rule
    + @PreAuthorize defence-in-depth). Validates transitions in OrderService.
  - Frontend OrderForm.jsx: orders table shows Order #, Date, Items summary,
    Total Due, and Status with colour-coded badges.
    MERCHANT sees "My Orders"; MANAGER/ADMIN see "All Orders" + merchant
    column + status action buttons (Processing, Dispatched, Cancelled).
  - Auto-polling: orders table refreshes every 15 seconds via setInterval.
  - Manual Refresh button for on-demand updates.
  - Status display mapping: ACCEPTED -> "Accepted", PROCESSING -> "Processing",
    DISPATCHED -> "Dispatched", CANCELLED -> "Cancelled". Legacy PENDING and
    CONFIRMED map to "Accepted" for display.

NOTE ON "REAL-TIME":
  WebSocket push is out of scope for this prototype. Polling every 15 seconds
  provides sufficient responsiveness for the tracking use case.

 ORD-US3 — Financial Balance Oversight
------------------------------------------------------------------------------

Full story text: section 4 — User Story 3.

STATUS: DONE

DONE:
  - GET /api/merchant-financials/balance (MERCHANT only, @PreAuthorize +
    SecurityConfig hasRole("MERCHANT")).
  - Returns BalanceResponse: outstandingTotal (sum invoices totalDue - sum
    payments), currency ("GBP"), oldestUnpaidDueDate, daysElapsedSinceDue.

    unpaid invoice's dueDate (ChronoUnit.DAYS.between).
  - Frontend: Invoices.jsx shows a balance summary card for MERCHANT users
    with outstanding total, due date, and elapsed days.
